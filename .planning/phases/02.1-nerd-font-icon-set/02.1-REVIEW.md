# Phase 02.1: Code Review Report

**Reviewed:** 2026-05-28
**Depth:** standard
**Files Reviewed:** 5
**Status:** issues_found

## Summary

Phase 02.1 wired a dual-table icon resolver (`_icon_to_glyph`), a 28-slot moon-phase
table, semantic ANSI color constants, and a render-time glyph selector behind a single
`icon_set` config toggle. The core mechanics are sound: the never-crash discipline is
maintained throughout, the backward-compat legacy-`icon` path is correctly handled, and
the `_moon_phase_index` clamping is correct.

Two blockers were found. The first is a logic bug in `_NERD_SUN_GLYPHS`: the set
includes `_WI_DAY_PARTLY`, which causes any partly-cloudy night condition to return a
moon-phase glyph instead of the defined `_WI_NIGHT_PARTLY` constant — making that
constant dead code. The second is a test fixture defect: three render tests in
`test_weather_fetch.py` assert for emoji sun glyphs (`🌅`/`🌇`) but run with no
`display` key in their `cfg`, so in the nerd default they will fail as soon as `astral`
is installed in the test environment.

## Critical Issues

### CR-01: `_NERD_SUN_GLYPHS` incorrectly includes `_WI_DAY_PARTLY` — partly-cloudy nights show moon phase

**File:** `claude-statusline.py:534`

**Issue:** `_NERD_SUN_GLYPHS` is defined as:

```python
_NERD_SUN_GLYPHS: frozenset = frozenset({_WI_DAY_CLEAR, _WI_DAY_PARTLY})
```

`_icon_to_glyph` (line 565) uses this set to decide whether to substitute the live
moon-phase glyph for a night URL. Because `_WI_DAY_PARTLY` is included, any
partly-cloudy or mostly-clear condition at night (NWS tokens: `sct`, `few`, `mostly
clear`, `mostly sunny`, `partly cloudy`, `partly sunny`) resolves to a moon-phase
glyph instead of the dedicated `_WI_NIGHT_PARTLY` constant that was defined precisely
for this case.

Affected inputs (non-exhaustive):
- textDescription `"Partly Cloudy"` + URL `…/night/sct`
- textDescription `"Mostly Clear"` + URL `…/night/few`
- keyword `"scattered"` + any night URL

`_condition_category` (line 622) has the same guard and will return `"moon"` for
partly-cloudy night conditions, causing them to render without ANSI color (RESET)
instead of the expected YELLOW for sun-dominant sky states.

`_WI_NIGHT_PARTLY` (defined at line 330, tested for existence in `test_nerd_icons.py`)
is completely unreachable in the current resolver — it is dead code.

**Fix:** Remove `_WI_DAY_PARTLY` from `_NERD_SUN_GLYPHS` so only fully-clear conditions
trigger the moon-phase substitution. Handle night-partly conditions explicitly inside the
partly entry match:

```python
# _NERD_SUN_GLYPHS: only FULLY CLEAR day glyphs get replaced by moon at night.
_NERD_SUN_GLYPHS: frozenset = frozenset({_WI_DAY_CLEAR})
```

Then in `_icon_to_glyph`, within the partly-entry match branch, add a separate
night-glyph swap before the moon-phase check:

```python
# In the partly entry match (after checking is_night and glyph in _NERD_SUN_GLYPHS
# would no longer trigger for _WI_DAY_PARTLY):
if is_night and glyph == _WI_DAY_PARTLY:
    return _WI_NIGHT_PARTLY
```

Apply the same fix to `_condition_category` so partly-cloudy night returns `"sun"` (or
a new `"cloud_night"` category) rather than `"moon"`.

### CR-02: `test_weather_fetch.py::TestWeatherSegmentRender` — three tests assert emoji sun glyphs but run with nerd-default `icon_set`

**File:** `tests/test_weather_fetch.py:701,804,816`

**Issue:** `TestWeatherSegmentRender.setUp()` builds `self.cfg` without a `"display"`
key (lines 612–620). `_weather_segment` defaults `icon_set` to `"nerd"` when the key
is absent (line 1429), and `_sun_segment` uses the same default (line 1337). With
`icon_set="nerd"`, `_sun_segment` returns `_WI_SUNRISE` or `_WI_SUNSET` (Nerd Font
PUA glyphs), not the emoji codepoints `🌅` (`\U0001f305`) or `🌇` (`\U0001f307`).

Three test methods assert `"\U0001f305" in inner or "\U0001f307" in inner`:

| Line | Test method |
|------|-------------|
| 701  | `test_fresh_with_pop_shows_all_three_chunks` |
| 804  | `test_beyond_ceiling_drops_to_sun_only` |
| 816  | `test_cold_cache_shows_sun_only` |

These tests currently pass only because `astral` is not installed in the CI/test
environment (causing them to be `skipTest`). Once `astral` is installed they will fail
with assertion errors.

**Fix:** Pin `icon_set="emoji"` in `TestWeatherSegmentRender.setUp()` (mirror the
approach used in `TestWeatherSegmentAlertOverride.cfg_with_emoji`, line 619 of
`test_weather_alerts.py`):

```python
self.cfg = {
    "location": {"lat": 35.4676, "lon": -97.5164},
    "weather": {"contact_email": "test@example.com", "show_weather": True},
    "units": {"temp_unit": "F"},
    "cache": {"weather_ttl": 600, "alerts_ttl": 300,
              "weather_max_stale": 3600, "alerts_max_stale": 900},
    "toggles": {"show_thinking_glyph": True},
    "thresholds": {"warn": 70, "crit": 90},
    "display": {"icon_set": "emoji"},   # <-- add this line
}
```

### IN-01: `from datetime import date as _date` is an inline import inside a render-path hot loop

**File:** `claude-statusline.py:568`

**Issue:** Inside `_icon_to_glyph`, within a nested `try` block, there is:

```python
from datetime import date as _date
phase = _moon_phase(_date.today())
```

Python caches module imports so this has no runtime cost after the first call. However,
it is inconsistent with the pattern used everywhere else (top-level `from datetime
import datetime, timedelta` at line 70), and `date` is not imported at module level.
The local alias `_date` is used only in this one call, making the intent less obvious.

**Fix:** Add `date` to the module-level datetime import:

```python
from datetime import datetime, timedelta, date as _date_cls
```

Then update the call site to use the module-level name. Alternatively, extend the
existing import:

```python
from datetime import datetime, date, timedelta
```

and call `_moon_phase(date.today())` directly without the local alias.

### IN-02: Precip chunk (`🌧️<pop>%`) always uses the rain emoji regardless of `icon_set`

**File:** `claude-statusline.py:1477`

**Issue:** The precip chunk is hardcoded to the rain emoji:

```python
precip_chunk = f"\U0001f327️{int(pop)}%"  # 🌧️
```

This means `icon_set="nerd"` produces a mixed bar: the condition glyph uses a Nerd Font
PUA codepoint, but the precipitation probability indicator remains an emoji.

The phase specification lists the four converted segments as "weather conditions/alerts,
sun events, thinking indicator, rate-limit" — the precip chunk is not in that list,
so this is consistent with the stated scope. However, a reader inspecting the nerd-mode
output will see an inconsistent mix that may or may not be intentional.

**Fix (if desired in a future phase):** Add a `nerd`-path precip glyph constant (e.g.,
`_WI_RAIN_SHOWERS` or `_WI_RAIN`) and select it via `icon_set`:

```python
if icon_set == "nerd":
    precip_glyph = _WI_RAIN
else:
    precip_glyph = "\U0001f327️"  # 🌧️
precip_chunk = f"{precip_glyph}{int(pop)}%"
```

No action needed now if the spec intentionally excludes the precip chunk.
