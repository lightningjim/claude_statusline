## Summary

Reviewed the single changed file `tests/test_gsd_segment.py` for the 05.1 test-only
hermeticity fix. The change restamps the handoff `timestamp` to `now()` inside
`TestGsdSegmentBuilder._call.fake_read_gsd_state` (copy-before-mutate) and adds one
regression guard test `test_builder_live_at_any_walltime`. Production source was not
modified (confirmed: the diff touches only the test file).

The core fix is **correct**. I traced the consumer chain:
`_call` -> `fake_read_gsd_state` -> `_gsd_segment` -> `_infer_gsd_lifecycle`
(`claude-statusline.py:1616-1664`). The production code reads `handoff["timestamp"]`,
computes `age = now - ts`, and sets `handoff_live = age <= _GSD_HANDOFF_STALE_SECONDS`
(`:1624`). The fixed fixture timestamp `2026-05-29T20:00:00.000Z` would age out of the
1-hour window (`_GSD_HANDOFF_STALE_SECONDS = 3600`, `:1374`) and silently flip the
executing tests into the idle/DIM fallback as wall-clock advanced — exactly the leak
this phase fixes. Restamping to `now()` removes the wall-clock dependency.

**Copy-before-mutate is sound.** `live_handoff = dict(handoff)` is a shallow copy, but
the mutation only reassigns the top-level scalar key `timestamp` (a string). No nested
list/dict in the fixture is touched, so a shallow copy is sufficient and the shared
module-level fixtures (`_VALID_HANDOFF`, `_BLOCKED_HANDOFF`, etc.) are never mutated.
All 62 tests pass; no fixed-timestamp assertions remain anywhere in the file that the
restamp could break.

The findings below are quality/robustness concerns, not correctness blockers.

## Warnings

### WR-01: Guard test adds no assertion beyond existing tests — weak as a "regression guard"

**File:** `tests/test_gsd_segment.py:897-910`
**Issue:** `test_builder_live_at_any_walltime` is the phase's headline deliverable, but
its assertions are a strict subset of the union of two pre-existing tests:
- `assertIn("05-02", result)` duplicates `test_executing_contains_plan_id` (`:891-895`)
- `assertIn("\033[32m", result)` duplicates `test_executing_contains_green` (`:912-916`)

Because `_call` *always* restamps now, this guard fires under exactly the same
conditions as those two siblings — if someone reintroduced a stale fixture timestamp
into `_call`, all three tests would fail identically. The docstring's claim that this
test "fails immediately rather than rotting silently" is therefore overstated: it has
no isolating mechanism (it does not, e.g., assert on the timestamp the fake produced,
nor freeze/advance the clock to prove wall-clock independence). As written it is a
redundant copy, not a distinct guard.
**Fix:** Make the guard actually exercise wall-clock independence so it earns its name.
Either (a) assert the negative — that the DIM idle escape is *absent* — to lock the
live/idle distinction the bug blurred:
```python
result = self._call(handoff=_VALID_HANDOFF)
self.assertIn("\033[32m", result)        # live executing glyph
self.assertNotIn("\033[2m", result)      # NOT the stale->idle DIM fallback
```
or (b) parametrize the fake's stamped age and assert that a *fixed past* timestamp
flips the segment to DIM while `now()` keeps it GREEN, proving the restamp is what
matters rather than re-asserting the happy path.

## Info

### IN-01: Restamp idiom duplicated across helpers instead of shared

**File:** `tests/test_gsd_segment.py:845-849` (and `:426-429`, `:461-464`, `:476-479`, `:638-641`, `:674-676`, `:692-695`, `:719-721`, `:1013-1014`, `:1027-1028`)
**Issue:** The `datetime.now(timezone.utc).strftime("%Y-%m-%dT%H:%M:%S.000Z")` restamp
plus `dict(handoff)` copy is now hand-rolled in ~10 places across two test classes.
The 05.1 change adds one more copy of it. Drift between copies (e.g., one forgetting
the copy, or using a different format the production `fromisoformat` parser tolerates
differently) is the exact failure class this phase exists to fix.
**Fix:** Extract a module-level helper, e.g.
`def _live_handoff(base): h = dict(base); h["timestamp"] = _now_z(); return h`, and
route `fake_read_gsd_state`, `_make_live_state`, and the inline restampers through it.
This is a test-maintainability improvement, not a correctness issue.

### IN-02: Inline `from datetime import ...` inside the fake instead of module top

**File:** `tests/test_gsd_segment.py:845`
**Issue:** `from datetime import datetime, timezone` is imported inside
`fake_read_gsd_state` on every call. This mirrors the existing style elsewhere in the
file (the sibling helpers do the same), so it is consistent, but it is needless repeated
import work and obscures the module's dependency surface.
**Fix:** Hoist `from datetime import datetime, timezone` to the top-of-file import block
alongside `json`, `os`, `sys`. Cosmetic; harmless as-is.
