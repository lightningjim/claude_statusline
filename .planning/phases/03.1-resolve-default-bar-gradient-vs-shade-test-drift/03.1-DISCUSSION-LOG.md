## Pre-discussion finding (root cause)

Before asking the decision, codebase analysis reframed the phase. The roadmap stated *"the default
now renders a gradient bar."* This is false at the code level: `DEFAULTS["bar_style"]`
(`claude-statusline.py:169`) is already `"shade"`. The two failing tests call `run_script()` without
isolating config, so they read the developer's live config
(`~/.claude/claude-statusline/claude-statusline.toml:34` → `bar_style = "gradient"`) and render the
personal gradient bar. The sibling `TestBarStylePresets` tests pass because `_run_with_toml`
(`tests/test_bottom_line.py:416`) backs up / restores the real config. Conclusion presented to user:
this is a **test-isolation defect**, not a code/intent drift.

| Option | Description | Selected |

**Notes:** Preserves Phase-3 D-09 (zero visual change for existing installs). Gradient remains the user's personal config choice only.

## Claude's Discretion

- Exact isolation mechanism (remove real config vs empty temp config) that genuinely drives `load_config`'s `DEFAULTS` fallback.
- Shared helper / fixture vs per-test isolation, and the precise default-assertion style (explicit `▓`/`░` counts vs "no gradient glyphs + counts").
- Scope of the sibling-audit hardening so long as default-asserting full-run tests no longer read the live config.

## Deferred Ideas

- Changing the shipped default to `gradient` — considered, rejected to preserve D-09 and the single-user/personal-tooling constraint; revisit only if general distribution or a default-look refresh is ever in scope.
