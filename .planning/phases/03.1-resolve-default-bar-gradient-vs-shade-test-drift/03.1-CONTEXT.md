# Phase 03.1: Resolve default bar gradient vs shade test drift - Context

**Gathered:** 2026-05-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Resolve the test/code drift surfaced in the Phase-3 default progress bar: two tests in
`tests/test_bottom_line.py` fail because the bar they observe is a gradient (`█▍…`) rather than
the asserted `▓` shade cell. Decide the intended shipped default, then make the default and the
tests agree. Scope is strictly the default-bar decision + realigning the affected tests (plus a
config-leak audit of sibling full-run tests). No new bar styles, no rendering changes.

</domain>

<decisions>
## Implementation Decisions

### Root cause (the real defect)
- **D-01:** The drift is a **test-isolation defect, not a code/intent drift.** `DEFAULTS["bar_style"]`
  in `claude-statusline.py:169` is already `"shade"` — Phase-3 D-09 ("zero visual change for existing
  installs") is intact. The two failing tests call `run_script()` directly, which executes the real
  script; the script reads the user's **live** config at
  `~/.claude/claude-statusline/claude-statusline.toml`, whose line 34 is `bar_style = "gradient"`.
  The tests therefore render the user's personal gradient bar and fail their `▓` assertions. The
  sibling tests in `TestBarStylePresets` pass precisely because `_run_with_toml`
  (`tests/test_bottom_line.py:416`) backs up / writes / restores the real config — the two failing
  tests skipped that isolation.

### Intended default
- **D-02:** The shipped default `bar_style` **stays `"shade"`.** `DEFAULTS` is unchanged; D-09 is
  preserved; fresh installs are visually unchanged. The user's preference for gradient lives in their
  personal config, **not** in the shipped default. (Changing the default to gradient was explicitly
  considered and rejected — see Deferred.)

### Test fix
- **D-03:** Realign the two named tests by **isolating them from the real config** and asserting the
  true no-config shade default (1 `▓` + 19 `░` at the fixture's 7%):
  - `tests/test_bottom_line.py::TestBottomLineFixture::test_bottom_line_bar_fill_cells`
  - `tests/test_bottom_line.py::TestBarStylePresets::test_default_no_config_shade_unchanged`
  Isolation must faithfully exercise `load_config`'s `DEFAULTS` fallback (no/empty config present
  during the run), mirroring `_run_with_toml`'s backup→run→restore-in-`finally` so the user's live
  config is never disturbed.
- **D-04:** **Audit sibling full-run tests** (any test invoking `run_script()` across the test suite)
  for the same config-leak vulnerability, and harden any that assert default/baseline behavior
  without config isolation. Goal: this class of drift cannot recur silently.

### Claude's Discretion
- Exact isolation mechanism — temporarily removing the real config vs writing an empty temp config —
  whichever genuinely drives the `DEFAULTS` fallback path in `load_config`. A shared helper
  (e.g. `_run_with_no_config`) or a `setUp`/fixture guard is preferred over re-skipping isolation
  per test.
- Whether to assert the shade default by explicit `▓`/`░` counts or by "no gradient glyphs present
  + correct counts" — pick the clearer, less brittle assertion.
- Scope of the D-04 audit's hardening (shared fixture vs per-test) so long as default-asserting
  full-run tests no longer read the user's live config.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `TestBarStylePresets._run_with_toml` (`tests/test_bottom_line.py:416`) — config-isolation pattern (backup real config → write temp → run → restore in `finally`). The fix should reuse/adapt this, with a "no/empty config" variant for the default-assertion tests.

### Established Patterns
- Full-run tests use `run_script()` (`tests/test_bottom_line.py:34`), which spawns the real `claude-statusline.py`. The script's `load_config` reads `~/.claude/claude-statusline/claude-statusline.toml` and deep-merges over `DEFAULTS`. **Any test asserting "default"/no-config behavior must isolate the real config or it inherits the developer's live settings.**
- Graceful-degradation contract (RUN-02): unknown/missing config silently falls back to `DEFAULTS` — the behavior the corrected tests should actually verify.

### Integration Points
- `claude-statusline.py:169` (`DEFAULTS` bar_style) is the single source of the shipped default; the corrected tests assert exactly this value's render output (shade).

</code_context>
