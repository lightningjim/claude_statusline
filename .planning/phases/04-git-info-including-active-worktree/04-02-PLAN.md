<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| stdin (session JSON) → `_git_segment` → git subprocess | `workspace.current_dir` / `cwd` from stdin selects the repo dir passed to `git -C` |
| git stdout (branch/worktree names) → rendered bar | text produced by git is printed into the status line |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05 | Tampering / EoP | dir from stdin reaches git via _git_segment → _run_git | mitigate | Inherited from Plan 01: _run_git uses fixed argv, no shell=True; _git_segment never concatenates the dir into a shell string. cfg/data accesses are all dict-guarded. |
| T-04-06 | Denial of Service | _git_segment runs two git calls every render (D-07 no cache) | accept (bounded) | Each call is timeout-guarded by _run_git (0.15s); worst case both omit and the segment returns None — the rest of the bar renders. No cache by design (D-07). |
| T-04-07 | Tampering (spoofing) | branch/worktree name with ANSI/control chars in the bar | accept | LOW risk, single-user tool; names are printed inert, never re-executed. Control-char stripping remains optional/deferred. |
| T-04-08 | Info Disclosure | git segment reveals branch/worktree of any current_dir | accept | By design — the segment's entire purpose is to surface the session's own repo state on the user's own terminal. No third party. |
</threat_model>
