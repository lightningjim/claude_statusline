## Warnings

### WR-01: Statement/Other alert silently drops its BOLD/DIM intensity (RESET cancels the modifier)

**File:** `claude-statusline.py:1186-1196` (`_alert_color`)
**Issue:** `_alert_color` returns `f"{intensity}{hue}"`. For the `Statement/Other` class the hue
is `RESET` (`\x1b[0m`). An `Immediate + Observed` Statement/Other alert therefore produces
`BOLD + RESET` = `\x1b[1m\x1b[0m`, where the trailing `RESET` immediately cancels the `BOLD`.
A `Future`/`Possible` Statement/Other likewise produces `DIM + RESET`, dropping the DIM. So the
D-06 intensity band is silently lost for the entire Statement/Other class — the alert renders
identically regardless of urgency/certainty. This is a fidelity bug for a meteorologist user:
an "Immediate/Observed" statement looks the same as a "Future/Possible" one. Verified directly:
`f"{BOLD}{RESET}"` → `'\x1b[1m\x1b[0m'`.
**Fix:** Use a non-resetting neutral hue for Statement/Other (e.g. a plain white/default SGR that
does not clear attributes), or reorder so intensity is re-applied after the hue. Minimal option:
map Statement/Other to `""` (empty hue) so intensity survives and the segment-level trailing
`RESET` still closes the run:
```python
hue_map = {
    "Warning":         RED,
    "Watch":           YELLOW,
    "Advisory":        CYAN,
    "Statement/Other": "",   # neutral; do NOT use RESET here (it cancels intensity)
}
```
Note the existing test `test_statement_alert_not_red` only asserts RED is absent and the string
is non-empty — it does not catch the cancelled intensity, so this regressed silently.

### WR-02: Sanitized event can collapse to empty, emitting a bare glyph with a dangling space

**File:** `claude-statusline.py:2599-2603` (`_weather_segment` alert override)
**Issue:** `safe_event` strips every non-printable / ESC character. If NWS `event` is present but
consists entirely of stripped characters (e.g. an all-control-char or non-Latin-unprintable
payload), `safe_event` becomes `""`, and `detail = f"{class_glyph} {safe_event}"` renders as
`"<glyph> "` with a trailing space and no event text. Per D-10 (omit-not-fake), a fragment with
no real content should be omitted or fall back, not rendered as a glyph plus whitespace. Verified:
`''.join(...)` on `"\x1b\x1b\x1b"` yields `''`.
**Fix:** Guard for an empty post-sanitization event and fall back (e.g. to the class name or to
the sun event):
```python
safe_event = "".join(
    ch for ch in str(event)
    if ch == " " or (ch.isprintable() and ch != "\x1b")
).strip()[:64]
if not safe_event:
    safe_event = best_class  # or skip the override entirely → sun fallback
detail = f"{class_glyph} {safe_event}"
```

### WR-03: `select_alert` tie-break maps omit valid CAP enum values, mis-ranking edge cases

**File:** `claude-statusline.py:1155-1158` (`select_alert._composite_key`)
**Issue:** The urgency/certainty tie-break dicts are incomplete relative to the CAP vocabulary:
urgency is `{"Immediate","Expected","Future"}` but CAP also defines `Past` (and `Unknown`);
certainty is `{"Observed","Likely","Possible"}` but CAP also defines `Unlikely` (and `Unknown`).
Both `.get(..., 0)` to 0, so `Past` urgency ranks equal to `Future`, and `Unlikely` certainty
ranks equal to `Possible`. This only affects ordering among alerts already tied on class and
severity, so impact is low — but for a meteorologist the operational ordering of two same-class
same-severity hazards can be wrong (a `Likely` should outrank an `Unlikely`, here they can tie or
invert relative to `Possible`). Note also this mapping is inconsistent with `_alert_intensity`,
which treats `Unlikely` as a DIM (low) signal — two different rank notions of the same field.
**Fix:** Complete the maps and align the certainty ordering with `_alert_intensity`:
```python
urg  = {"Immediate": 3, "Expected": 2, "Future": 1, "Past": 0}.get(
            props.get("urgency", "Unknown"), 0)
cert = {"Observed": 3, "Likely": 2, "Possible": 1, "Unlikely": 0}.get(
            props.get("certainty", "Unknown"), 0)
```

### WR-04: Per-class glyph resolution duplicated between override block and `_build_alert_tally`

**File:** `claude-statusline.py:2594-2597` (override) and `claude-statusline.py:1222-1225`
(`_build_alert_tally`)
**Issue:** The `icon_set`-branched glyph lookup
(`_ALERT_CLASS_GLYPHS_NERD.get(cls, _WI_ALERT_STATEMENT)` vs `_ALERT_CLASS_GLYPHS_EMOJI.get(cls, "ℹ️")`)
is hand-duplicated in two places. If a future change alters the fallback glyph or adds a class,
the two paths can drift, producing a primary glyph that disagrees with the tally glyph for the
same class. This is a maintainability/consistency hazard, not a current behavioral bug.
**Fix:** Extract a single helper and call it from both sites:
```python
def _alert_class_glyph(cls: str, icon_set: str) -> str:
    if icon_set == "nerd":
        return _ALERT_CLASS_GLYPHS_NERD.get(cls, _WI_ALERT_STATEMENT)
    return _ALERT_CLASS_GLYPHS_EMOJI.get(cls, "ℹ️")
```

## Info

### IN-01: `_classify_alert_class` event-name fallback is case- and substring-sensitive

**File:** `claude-statusline.py:982-985`
**Issue:** The fallback `event.endswith(word)` for `("Warning","Watch","Advisory")` is exact-case
and exact-suffix. NWS event names are consistently title-cased so this works for the real feed,
but a lowercased or trailing-punctuation event (e.g. from a non-standard source or a future feed
change) would fall through to `Statement/Other`. Acceptable given VTEC is the primary path and
this is only the fallback, but worth a defensive `.title()`/case-fold if event sources ever vary.
**Fix:** Optional — `for word in (...): if event.lower().rstrip(".").endswith(word.lower()): return word`.

### IN-02: Incomplete tie-break maps in `select_alert` use bare magic numbers

**File:** `claude-statusline.py:1155-1158`
**Issue:** The inline `{"Immediate": 2, ...}` and `{"Observed": 2, ...}` literals are magic-number
ranking tables embedded in a closure, parallel to the module-level `_SEVERITY_RANK` /
`_ALERT_CLASS_RANK` constants. For consistency and discoverability they should be module-level
named constants (`_URGENCY_RANK`, `_CERTAINTY_RANK`) like the other two rank tables.
**Fix:** Promote to module-level dicts alongside `_SEVERITY_RANK` and reference them.

### IN-03: `_alert_intensity` docstring lists a band the code does not implement

**File:** `claude-statusline.py:996-998`
**Issue:** The docstring's middle band reads "Expected / Likely → '' (normal)", but the code only
returns `""` as a final fallthrough for *anything not matched* by the BOLD or DIM branches — there
is no explicit `Expected`/`Likely` check. Behavior is correct, but the docstring implies a check
that isn't there, which can mislead a future maintainer adding cases.
**Fix:** Reword the comment to "anything not Immediate+Observed and not Future/Possible/Unlikely →
normal" or add the explicit branch for clarity.

### IN-04: Module-level test header docstring describes the pre-reshape `select_alert` contract

**File:** `tests/test_weather_alerts.py:8-10`
**Issue:** The file's top docstring still states `select_alert(survivors)` returns
