# Phase 07.1 Plan 02: Prune live_ids refined for resolved-but-degraded retention Summary

**One-liner:** `live_ids` prune in `fetch_claude_status` refined to retain dismissed resolved-but-still-degraded incident ids (D-06), with a tracked-component status map gate; truly-stale dismissals still prune; full suite 219 passed.

## What Was Built

This plan fixed **PATTERNS.md Risk #1**: the `fetch_claude_status` prune block previously dropped a dismissed incident's id from `live_ids` the moment the incident resolved, allowing it to re-surface green (dismissed → pruned → unmuted) — violating D-06 ("muting wins in every state").

**Task 1 — Refined `live_ids` predicate in `fetch_claude_status` (D-06):**

The `live_ids` build now has two rules:

- **Rule (a) — unchanged**: unresolved incidents (`investigating`, `identified`, `monitoring`) are always included in `live_ids`.
- **Rule (b) — new**: a `resolved` incident id is added to `live_ids` when at least one of its tracked components is still non-operational in the live feed. This retains the dismissal for exactly as long as the incident is capable of driving the segment (D-01 lifecycle).

A resolved incident whose tracked components are all back to `operational` stays OUT of `live_ids` — its dismissal prunes as a genuinely stale entry (consistent with the WR-03 intent of not deferring prune indefinitely).

Implementation details:
- A tracked component name→status map (`_prune_comp_map`) is built from `summary["components"]` before the incident loop, mirroring the normalization in `_derive_claude_status` (~:1748-1759): only names in `_CLAUDE_TRACKED_COMPONENTS`, default `"operational"` for missing status.
- The per-incident loop now uses `isinstance` guards consistently and extracts the string-id guard to the top of the loop (no behavior change for the unresolved path).
- The WR-03 comment is rewritten to document the D-06 resolved-but-degraded retention rule and explain why resolved-and-operational incidents still prune.
- The inner `try/except: dismissals = {}` (D-10) and the `write-only-if-changed` branch are preserved verbatim.
