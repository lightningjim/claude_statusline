## Fixed Issues

### CR-01: Unexplained-degraded component masked GREEN by a resolved incident on a different component

**Files modified:** `claude-statusline.py`
**Commit:** d196661
**Applied fix:** Restructured `_derive_claude_status` Rule 3. It no longer returns on the
first degraded component in sort order. It now classifies EVERY degraded tracked component
first: if ANY degraded component has no non-suppressed resolved incident explaining it, that
RED unexplained component wins immediately (highest-severity per the existing `severity_map`),
regardless of any resolved-explained component found earlier in sort order. The GREEN
"resolved" verdict is emitted only when every degraded component is explained by a
non-suppressed resolved incident (newest `updated_at` among the resolved wins). Rule 1 >
Rule 2 > Rule 3 ordering (D-04) and the body-level try/except never-raise (D-10) are
preserved. Verdict is now order-independent (verified via both component sort orderings in
the WR-03 tests).

### CR-02: Dismissed resolved item re-surfaces GREEN, kept alive by an unrelated incident (D-06 Risk #2)

**Files modified:** `claude-statusline.py`
**Commit:** b2acf73, 069198f
**Applied fix:** The render-time mute re-check in `_claude_status_segment` no longer keeps a
baked resolved/degraded section alive when "any" tracked incident survives suppression.
For a baked `kind in {"resolved","degraded"}` section it now binds the suppression decision
to the section's specific explaining incident id: it locates that incident in the cached
`tracked_incidents` and returns `None` if the incident is absent from the live feed OR is
suppressed at render time — never green, never red (D-06 "muting wins in every state"). For
baked `kind=="incident"` sections the existing first-non-suppressed override behavior is
preserved. A second commit (069198f) refined the derivation degraded branch so a component
whose only explanation is a MUTED incident bakes that muting incident's id (so render mutes
it to `None`), while a genuinely-unexplained degraded component keeps `incident_id=None` and
correctly stays RED. The resolved label still routes through the same Step-6 ANSI/non-
printable + width sanitizer as active titles; the segment stays inside its try/except (D-10).

### WR-01: `_derive_claude_status` resolved scan re-runs per degraded component (redundant + amplifies CR-01)

**Files modified:** `claude-statusline.py`
**Commit:** d196661
**Applied fix:** Resolved by the same CR-01 restructure. The Rule-3 loop now classifies each
degraded component (collecting the resolved match without an early return), removing the
scan-and-return-on-first pattern that enabled the masking bug. The `severity_map` was hoisted
out of the loop body.

### WR-02: Cache section omits the explaining-incident identity, forcing render-time guessing

**Files modified:** `claude-statusline.py`
**Commit:** a648a61 (payload), d196661 (derivation return-dict fields)
**Applied fix:** `_derive_claude_status` now returns `incident_id` and `component` on
degraded/resolved verdicts, and `fetch_claude_status` bakes both into the noteworthy cache
payload. The new fields are documented in the `_derive_claude_status` docstring. The render
path reads them defensively (missing field → safe `None` fallback → prior behavior), and
Rule 1 (incident) / Rule 2 (maintenance) returns intentionally omit them. This is the
binding that makes the CR-02 render fix possible.

### WR-03: Test coverage gap hides CR-01 and CR-02 (single-incident / single-component only)

**Files modified:** `tests/test_claude_status.py`
**Commit:** 97449f4
**Applied fix:** Added multi-component / multi-incident regression tests that fail against the
pre-fix code:
- `test_cr01_unexplained_degraded_wins_over_resolved_code_first` and
  `..._ai_first` — two degraded components (one resolved-explained, one unexplained) assert
  the verdict is RED for the unexplained component under BOTH component sort orderings.
- `test_cr01_all_degraded_components_resolved_returns_green` — boundary: all-explained →
  green resolved with a bound `incident_id`.
- `test_cr02_dismissed_explaining_incident_with_unrelated_survivor_returns_none` — baked
  resolved for A, dismiss only A, unrelated non-suppressed B present → segment is `None`.
- `test_cr02_unrelated_survivor_does_not_leak_when_explaining_present` — positive control:
  binding selects the explaining incident's label (A), never the unrelated survivor (B).
- `test_cr02_explaining_incident_absent_from_feed_returns_none` — explaining id gone from
  the feed → `None`.

Three pre-existing tests (`test_muted_resolved_returns_none_not_green`,
`test_muted_resolved_keyword_returns_none`, `test_muted_degraded_returns_none_not_red`) were
updated to bake the realistic `incident_id`/`component` fields now produced by derivation, so
they exercise the CR-02 binding contract rather than the obsolete "scan-all" shape. All
existing claude_status tests continue to pass.
