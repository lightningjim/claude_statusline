<objective>
Fix PATTERNS.md **Risk #1** — the `fetch_claude_status` `live_ids` prune block (≈:2103-2133) currently un-mutes a dismissed incident the moment it resolves, because it keys `live_ids` off UNRESOLVED incidents only. Under Phase 07.1, a resolved incident is still actively surfaced while its tracked component remains degraded (D-01 lifecycle); dropping its dismissal at resolution would let it re-surface as green `resolved:` — violating D-06 ("muting wins in every state").

Purpose: The dismissal must stay alive for exactly as long as the incident is still capable of driving the segment. This plan reconciles the WR-03 prune-aggressiveness comment with the new resolved-surfacing lifecycle so the auto-prune still clears truly-stale dismissals but never prematurely un-mutes a resolved-but-degraded incident.

Output: A refined `live_ids` predicate (or equivalent retention rule) in `fetch_claude_status`, an updated WR-03 comment, and a regression test proving a dismissed resolved-but-degraded incident stays muted across the prune.

Implements: D-06 (muting wins through the resolved phase) at the store-prune layer. This is the data-layer half of D-06; the render-layer half (Risk #2 fall-through guard) is Wave 3 Plan 03.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| status.claude.com feed → fetch/prune | Untrusted remote JSON drives which ids are considered "live"; runs ONLY in the detached `--refresh` child, never on the render path |
| tool-owned dismissal store (local JSON) → prune | Local file read/written by `_prune_dismissals` / `write_dismissals`; corrupt store must degrade to no-op |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07.1-05 | Tampering | feed-controlled incident status/components driving live_ids retention | mitigate | Retention is gated by membership in `_CLAUDE_TRACKED_COMPONENTS` (a fixed frozenset) AND a non-operational tracked status; a hostile feed can at most keep a user's OWN dismissal alive longer (fails safe toward staying muted), never inject a new dismissal or delete one. String-id guard retained. |
| T-07.1-06 | Repudiation / Integrity | dismissal store mutation during prune | mitigate | Write-only-if-changed preserved; the prune runs inside its own `try/except` that sets `dismissals={}` on any error so a corrupt/oversized store never corrupts the fetch or crashes. Tool never writes the user's TOML (D-05). |
| T-07.1-07 | Denial of Service | oversized incidents[] / components[] in feed | mitigate | Single linear pass building a bounded tracked component map (≤3 tracked) + one pass over incidents; per-item `isinstance` guards; no recursion/regex. Whole `fetch_claude_status` body in `try/except: pass` (D-10). Runs in the detached child — cannot block the render path regardless. |
| T-07.1-SC | Tampering | npm/pip/cargo installs | accept | No new dependencies introduced; stdlib-only edit to an existing function. No install tasks in this phase. |
</threat_model>
