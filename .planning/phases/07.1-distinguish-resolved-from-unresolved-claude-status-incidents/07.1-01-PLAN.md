<objective>
Establish the resolved-incident DERIVATION + CACHE contract (Wave 1). This plan teaches `_derive_claude_status` Rule 3 to distinguish a resolved-but-still-degraded component from an unexplained outage (D-04/D-05/D-01), adds the GREEN `resolved` hue to `_claude_status_color` (D-03), and widens `_collect_tracked_incidents` to carry resolved incidents into the cache (D-06/D-07 enabler). Purpose: The discretion note in CONTEXT (D-03/D-05/D-06) directs the resolved-vs-red decision into derivation so the cached payload carries `kind="resolved"` and render-time suppression re-applies exactly as for active incidents. This plan creates that cache shape so downstream waves have a stable contract.

Output: A `resolved` severity/kind flowing through `_derive_claude_status`, a GREEN mapping in `_claude_status_color`, resolved entries in `_collect_tracked_incidents`, a new fixture, and unit tests — all green.

Implements: D-01 (lifecycle keyed to live component state), D-02 (same tracked-component gate), D-03 (GREEN color), D-04 (Rule order unchanged, only Rule 3 rendering changes start here), D-05 (green only when a resolved incident touches that component, else red), D-06 (resolved-incident scan respects _is_suppressed).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| status.claude.com feed → derivation | Untrusted remote JSON (incident titles, component names, statuses, timestamps) parsed in `_derive_claude_status` / `_collect_tracked_incidents` |
| local cache file → (downstream Wave 3 render) | RAW (unsanitized) labels are stored in the cache by this plan; sanitization is the render/print edge's job |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07.1-01 | Tampering | feed incident title / component name (ANSI / control-char injection) | accept (here) / mitigate (Wave 3) | Labels are stored RAW in cache by design (RAW-in-cache, sanitize-at-edge). No terminal output happens in this plan. The ESC/non-printable strip + width-bound sanitizer is re-applied at render (Wave 3 Plan 03) and at print (`--status-incidents`). No new raw-render sink is introduced here. |
| T-07.1-02 | Denial of Service | resolved-incident scan over `summary["incidents"]` (oversized / deeply-nested feed) | mitigate | Scan iterates the same already-fetched in-memory list Rule 1 iterates; per-item `isinstance` guards skip non-dicts; no recursion, no regex on the derivation path here (title matching/ReDoS cap lives in `_is_suppressed`, reused unchanged). Bounded O(incidents × tracked) work. |
| T-07.1-03 | Denial of Service | corrupt / huge / non-dict summary input | mitigate | `_derive_claude_status` and `_collect_tracked_incidents` retain their whole-body `try/except` returning `None`/`[]`; a resolved-scan failure degrades to the existing red/unresolved behavior. Explicit does-not-raise test added (Task 2). |
| T-07.1-04 | Elevation of Privilege | suppressed resolved incident surfacing as a benign green item | mitigate | The Rule 3 resolved scan reuses the SAME `_is_suppressed(...)` gate Rule 1 uses (single ordering, `_CLAUDE_IMPACT_RANK`); a muted resolved incident is never matched, so it cannot produce the resolved item (D-06). Render-time re-check (Wave 3) is the second enforcement layer. |
</threat_model>
