# Phase 07.1: Distinguish resolved from unresolved Claude status incidents - Context

**Gathered:** 2026-06-18
**Status:** Ready for planning

<domain>
## Phase Boundary

Inside the existing **Phase 6/7 Claude service-health segment**, make the bar
tell the truth about a **recently-resolved** incident instead of misrepresenting
it as a live red outage.

**Reported symptom:** after `status.claude.com` marks an incident resolved, the
affected tracked component can linger in a degraded state (e.g. `partial_outage`)
while its incident is already `resolved`. Today that fires Rule 3 of
`_derive_claude_status` and the bar shows a red **"Claude Code: partial outage"**
as if it were an active major outage — even though the incident is resolved
("the alert hasn't cleared yet"). The goal: **keep surfacing important
recently-resolved incidents but render them visibly as resolved** (non-alarming),
since Anthropic resolving it on their dashboard doesn't guarantee it won't return.

This phase **refines how the degraded-component case (Rule 3) renders** — it does
NOT add a new endpoint (resolved incidents already linger in `summary.json` for
days, per the existing `fetch_claude_status` comment), does NOT change the
Phase 6 tracked-component set or the active-incident / maintenance rules, and does
NOT change quiet-when-healthy or never-crash behavior.

**Out of scope:** incident-history TUI, notifications, remote/synced state,
changing the tracked-component set, changing active-incident or maintenance
derivation, new fetch endpoints. Those are separate capabilities.

</domain>

<decisions>
## Implementation Decisions

### Resolved lifecycle — what to surface and for how long
- **D-01:** **Show a resolved incident only while its affected tracked component
  remains non-operational; drop it the moment the component returns to
  `operational`.** The lifecycle is keyed to the live component state, not a
  fixed timer. This directly reframes the lingering-degraded symptom: while the
  component is still degraded but its incident is resolved, we show it as
  resolved; when the component flips back to operational, the segment goes quiet
  (Phase 6 quiet-when-healthy, D-01, preserved). If the component was already
  operational at resolution, nothing is shown (truthful — nothing degraded).
- **D-02:** **Same qualifying gate as active incidents:** a resolved incident
  qualifies only if it touches a tracked component (`Claude Code`, `claude.ai`,
  `Claude Cowork`) — identical to the active-incident rule. No separate impact
  floor and no "only ones we previously showed" filter.

### Resolved visual treatment
- **D-03:** **GREEN + check-circle glyph + `resolved:` prefix.**
  - Color: `GREEN` (the existing constant). Reads unmistakably as resolved/good
    and is distinct from active (red exclamation) and maintenance (dim wrench).
  - Glyph: the check-circle `_NF_GSD_DONE` (` ` nf-fa-check_circle, already in
    the repo) for the nerd-font set; an equivalent emoji fallback (e.g. ✅) for
    the emoji set (exact emoji = planning discretion, mirror existing fallbacks).
  - Label: **`resolved:` prefix** on the incident title, e.g. `resolved: API
    errors`. Word + glyph both convey the state so it survives in no-color /
    glyph-fallback terminals (truth-telling, [[statusline-omit-not-fake]]).
  - Must read as **resolved** — not faked-healthy (segment is still shown, not
    hidden) and not alarming-active (no red).

### Priority / ordering and the red-lag fix
- **D-04:** **Refine Rule 3 rendering only; do not reorder derivation.** The
  derivation priority stays: **active incident (Rule 1) > maintenance (Rule 2) >
  degraded component (Rule 3) > None.** Active incidents and maintenance are
  unchanged. Only Rule 3 changes *how it renders*.
- **D-05:** **Resolved-green replaces the red Rule-3 line ONLY when a resolved
  incident in the feed actually touches that degraded component.** A component
  degraded with **no matching resolved incident** stays **red** (unexplained
  outage = still alarming, unchanged). "Only resolved when we know what
  resolved" — never paint an unexplained outage benign green.

### Interaction with the Phase 7 filter
- **D-06:** **A muted incident stays fully muted through its resolved phase.**
  If an incident was suppressed while active (id-dismiss per D-03 of Phase 7, or
  keyword/regex per Phase 7 D-01/D-06), it produces **no green `resolved:`** when
  it later resolves. Critically, a degraded component explained **only** by a
  muted incident **stays quiet — it must NOT fall back to the red Rule-3 line.**
  Muting wins in every state (active or resolved); the bar never gets louder
  because something was muted. (Escalation re-surface, Phase 7 D-03, still applies
  to the *active* path; a resolving incident has falling impact, so the dismissal
  stands.)
- **D-07:** **`--status-incidents` lists resolved-but-still-degraded incidents
  too, with a STATE column** (active / resolved / dismissed / stale) so the CLI
  remains a complete picture of what is driving — or could drive — the bar.
  Extends the existing Phase 7 `--status-incidents` output shape.

### Claude's Discretion
- **Safety cap on lingering resolved-green:** if Anthropic never flips the
  component back to operational, whether a resolved entry self-clears after a max
  age vs relying solely on the existing `status_max_stale` (900s) cache ceiling.
  Default: rely on the cache ceiling + live component state (no extra timer)
  unless planning finds it insufficient.
- **Incident→component matching strictness:** default is tracked-component-name
  intersection, newest resolved incident wins when several match the same
  degraded component. Tightening (require impact/recency match) is discretion.
- Exact emoji-set fallback glyph for the resolved state (mirror existing emoji
  fallbacks; a ✅-class glyph).
- Where the resolved-vs-red branch is computed (inside `_derive_claude_status`
  Rule 3 so the cached payload already carries a resolved `kind`/`severity`, vs a
  render-time branch in `_claude_status_segment`). Lean toward computing in the
  derivation and threading a `kind="resolved"` (or equivalent) through the cache
  + the render-time suppression re-check (Phase 7 Plan 04), so muting (D-06) is
  re-applied at render exactly as it is for active incidents.
- Whether the cached `tracked_incidents` list (and `_collect_tracked_incidents`)
  must widen to include resolved-but-degraded incidents so render-time
  suppression (D-06) and `--status-incidents` (D-07) can see them — likely yes;
  confirm at planning.

</decisions>

<specifics>
## Specific Ideas

- The whole point (carried from Phase 7): "**lit = something I should actually
  care about**", extended here to "**and the color tells the truth about whether
  it's still happening.**" Red = active problem; green `resolved:` = it broke,
  it's fixed, the component just hasn't fully cleared; dim = planned maintenance.
- Truth-telling over convenience ([[statusline-omit-not-fake]]): a resolved
  incident is **shown and labeled**, never silently faked healthy while the
  component is still degraded, and never shown red once it's actually resolved.
- The fix is deliberately **minimal and consistent**: it changes only how the
  existing degraded-component case (Rule 3) renders, reusing existing color/glyph
  assets and the Phase 7 render-time suppression machinery.

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- **`GREEN` constant + `_NF_GSD_DONE` check-circle glyph** — directly satisfy the
  resolved visual (D-03); no new glyph/color needs to be authored for nerd-font.
- **`_is_suppressed` + the Phase 7 render-time suppression block** in
  `_claude_status_segment` — reuse verbatim so muting (D-06) applies to resolved
  items exactly as it does to active ones (single ordering, no duplication).
- **Rule 3 degraded-component branch** in `_derive_claude_status` — the single
  integration point for D-04/D-05 (compute resolved-vs-red there, thread the
  result through the cache).
- **`--status-incidents` printer + cached `tracked_incidents`** — extend for the
  STATE column (D-07).
- **`summary.json` resolved incidents linger in the feed** — the resolved signal
  is already fetched; no new endpoint (D-01).

### Established Patterns
- Builders/derivation return `None`/omit on bad data; never crash, never fake
  ([[statusline-omit-not-fake]]). A resolved-detection failure must degrade to the
  existing behavior, not raise.
- Render path reads cache + the local dismissal store only; never blocks on the
  network. The resolved branch must stay equally cheap and failure-tolerant.
- `cfg` threaded explicitly; glyphs via `icon_set`; severity→hue via
  `_claude_status_color`.

### Integration Points
- `_derive_claude_status` Rule 3 (D-04/D-05); `_claude_status_color` /
  severity-mapping for the GREEN resolved case (D-03); `_claude_status_segment`
  render + Plan-04 suppression re-check (D-03/D-06); `_collect_tracked_incidents`
  + cache payload (D-06/D-07); `--status-incidents` printer (D-07).
- Tests under `tests/` (closest: `tests/test_claude_status.py`) — add cases for:
  resolved incident + still-degraded component → green `resolved:`; degraded
  component with no matching incident → red (unchanged); component operational →
  quiet; active incident outranks resolved; maintenance outranks resolved-degraded
  (Rule-3 order kept); muted incident resolves → stays quiet, no red fallback
  (D-06); `--status-incidents` STATE column (D-07); malformed feed → no crash.

</code_context>

<deferred>
## Deferred Ideas

- Fixed time-window lifecycle for resolved incidents (show for N minutes after
  `resolved_at`) — considered and not chosen; lifecycle is keyed to live component
  state (D-01). Could return as a knob if the component-state approach proves
  insufficient.
- Encoding past severity in the resolved look (dim-red for a resolved major) —
  considered; rejected for a uniform GREEN resolved treatment (D-03) to avoid
  dim-red still reading as alarming.
- Resolved-degraded outranking maintenance — considered; kept Rule-3 order to
  minimize blast radius vs Phase 6 (D-04).
- Surfacing a resolved-green for a muted incident ("the thing you muted is fixed")
  — rejected; muting wins in every state (D-06).
- Incident-history TUI, notifications, remote/synced state, new fetch endpoints —
  out of scope (separate capabilities).

</deferred>
