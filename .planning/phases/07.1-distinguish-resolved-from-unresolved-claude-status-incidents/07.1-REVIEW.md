load; both are missed by the test suite because every relevant test uses a single incident /
single degraded component.

are all intact.

## Critical Issues

### CR-01: Unexplained-degraded component masked GREEN by a resolved incident on a *different* component

**File:** `claude-statusline.py:1868-1919` (`_derive_claude_status` Rule 3)

**Issue:**
Rule 3 iterates `for comp_name in sorted(_CLAUDE_TRACKED_COMPONENTS)` and `return`s on the
**first** non-operational component it encounters. When two tracked components are degraded
simultaneously — one explained by a resolved incident, one *unexplained* (a real, ongoing
outage with no incident at all) — the verdict depends entirely on alphabetical sort order of
the component names, not on severity. If the resolved-explained component sorts first, the
function returns `{"severity":"resolved","kind":"resolved"}` (GREEN) and the genuinely-broken,
unexplained component is never reached.

This violates the explicit review-focus invariant *"unexplained-degraded must stay red"* and
the project's omit-not-fake / truth-telling rule (`[[statusline-omit-not-fake]]`, D-10): the
bar tells the user everything is fixed (green) while a tracked service is actively down.

Reproduced from a fresh module load:

```python
summary = {
  "components": [
    {"name": "Claude Code", "status": "partial_outage"},  # explained by resolved incident
    {"name": "claude.ai",   "status": "partial_outage"},  # UNEXPLAINED — should stay RED
    {"name": "Claude Cowork","status": "operational"},
  ],
  "incidents": [
    {"id":"r1","name":"Code fixed","status":"resolved","impact":"major",
     "updated_at":"2026-06-18T12:00:00Z","components":[{"name":"Claude Code"}]},
  ],
  "scheduled_maintenances": [],
}
_derive_claude_status(summary)
# -> {'severity':'resolved','label':'Code fixed','kind':'resolved'}   (GREEN — WRONG)
```

Swapping which component carries the resolved incident flips the verdict to the correct red
degraded result, proving the outcome is order-dependent rather than intentional priority.

**Fix:**
A resolved verdict must never be returned while any tracked component is degraded *without* a
resolved explanation. Make Rule 3 collect across all degraded components first, and let red
(unexplained-degraded) win over resolved:

```python
resolved_result = None
for comp_name in sorted(_CLAUDE_TRACKED_COMPONENTS):
    status = comp_status.get(comp_name, "operational")
    if status not in non_operational:
        continue
    matched = _scan_resolved_for(comp_name)  # the existing inner try/except scan
    if matched is None:
        # unexplained-degraded → red wins immediately, regardless of other comps
        human_state = _CLAUDE_STATUS_LABELS.get(status, status.replace("_", " "))
        severity = severity_map.get(status, "minor")
        return {"severity": severity, "label": f"{comp_name}: {human_state}", "kind": "degraded"}
    # remember the resolved match but keep scanning for an unexplained-degraded comp
    if resolved_result is None:
        resolved_result = {"severity": "resolved", "label": matched.get("name",""), "kind": "resolved"}
if resolved_result is not None:
    return resolved_result
```

Add a regression test with two degraded components (one resolved-explained, one unexplained)
asserting `kind == "degraded"`, and run it for *both* sort orderings.

### CR-02: Dismissed resolved item re-surfaces GREEN, kept alive by an unrelated incident (D-06 Risk #2 violation)

**File:** `claude-statusline.py:3706-3744` (`_claude_status_segment` render-time suppression)

**Issue:**
The render-time suppression loop scans the **entire** `tracked_incidents` list and selects the
**first** non-suppressed entry as `surviving_inc`, then — for a baked `kind=="resolved"` (or
`"degraded"`) section — preserves the baked kind/severity/label and renders. There is no binding
between `surviving_inc` and the incident that actually *explains* the baked section (the cache
section stores no explaining-incident id or component — see `:2256-2262`). So a resolved item
whose own explaining incident has been dismissed is still rendered GREEN as long as *any other*
tracked incident is non-suppressed.

This breaks D-06 Risk #2 ("a baked resolved item explained ONLY by a muted incident must render
None — never red, never green") in the realistic flow: user runs `--dismiss A` between fetches;
at the next render incident A is suppressed but unrelated resolved incident B is not, so the
segment renders `resolved: <A's label>` in GREEN — surfacing the very incident the user muted,
with the wrong incident's title.

Reproduced from a fresh module load:

```python
sec = {
  "fetched_at": now-60, "noteworthy": True, "severity": "resolved",
  "label": "Claude Code fixed", "kind": "resolved",      # baked from incident A
  "tracked_incidents": [
    {"id":"A","impact":"major","status":"resolved","title":"Claude Code fixed","component":"Claude Code"},
    {"id":"B","impact":"minor","status":"resolved","title":"Unrelated AI thing","component":"claude.ai"},
  ],
}
# dismiss ONLY A
dismissals = {"A": {"impact_at_dismiss":"major","dismissed_at":now-10}}
_claude_status_segment({}, cfg)
# -> '\x1b[32m resolved: Claude Code fixed\x1b[0m'   (GREEN — should be None)
```

The existing test `test_muted_resolved_returns_none_not_green` (`tests/test_claude_status.py:1703`)
passes only because it uses a single-element `tracked_incidents=[resolved_inc]`, so there is no
second surviving incident — the multi-incident case is uncovered (see WR-03).

Note: the structurally-identical degraded variant pre-dates 07.1 (the loop existed in Phase 07),
but 07.1 *widened* `tracked_incidents` to include resolved entries and added the resolved render
branch, turning the latent mismatch into a user-visible GREEN "resolved" with a muted/wrong label.

**Fix:**
Bind the render-time suppression decision to the section's explaining incident, not to "any
surviving incident". Options:

1. Bake the explaining incident id (and/or component) into the cache section in
   `fetch_claude_status` (`:2256-2262`), then at render time only consult that specific entry's
   suppression state. If that entry is suppressed → return None. This is the cleanest and also
   fixes the wrong-label symptom.
2. For baked `kind in ("resolved","degraded")`, restrict the suppression scan to tracked
   incidents whose `component` matches the baked component (requires baking `component`), and
   return None if none of *those* survive.

Add a multi-incident regression test: baked resolved for A, dismiss A, leave unrelated B
non-suppressed, assert the segment is `None`.

## Warnings

### WR-01: `_derive_claude_status` resolved scan re-runs per degraded component (redundant + amplifies CR-01)

**File:** `claude-statusline.py:1871-1908`

**Issue:**
The resolved-incident scan (the inner `try/except` over `incidents_raw`) is nested inside the
`for comp_name` loop, so it re-walks the full incidents list once per degraded component. More
importantly, its placement inside the early-`return` loop is what enables CR-01. Hoisting the
scan logic and computing the verdict across all degraded components (per the CR-01 fix) removes
both the redundancy and the masking bug.

**Fix:** Restructure per CR-01 so the loop classifies every degraded component before deciding,
rather than scanning-and-returning on the first.

### WR-02: Cache section omits the explaining-incident identity, forcing render-time guessing

**File:** `claude-statusline.py:2255-2267` (`fetch_claude_status` payload build)

**Issue:**
The baked payload stores `severity/label/kind/tracked_incidents` but not which incident (or
component) produced the verdict. The render path therefore cannot tell which `tracked_incidents`
entry is the section's actual driver, which is the root enabler of CR-02 and forces the
"first non-suppressed wins" heuristic. Even absent CR-02, this makes the surviving-incident
label override (`:3741`) potentially attach an unrelated incident's title to the segment.

**Fix:** Add an `incident_id` (and optionally `component`) field to the noteworthy payload set
by `_derive_claude_status`/`fetch_claude_status`, and have the render path key suppression and
label resolution off that field. (Same change recommended in CR-02 fix option 1.)

### WR-03: Test coverage gap hides CR-01 and CR-02 (single-incident / single-component only)

**File:** `tests/test_claude_status.py:1703-1724` (`test_muted_resolved_returns_none_not_green`),
`tests/test_claude_status.py:383-510` (resolved-vs-red suite)

**Issue:**
Every resolved-path test uses exactly one tracked incident and/or one degraded component. The
suite never exercises (a) two degraded components where one is unexplained, or (b) a baked
resolved section with a dismissed explaining incident *plus* a second non-suppressed incident.
These are precisely the configurations where CR-01 and CR-02 manifest, so the green suite gives
false confidence. Per `[[user-precision-review]]`, this code needs concrete multi-incident
verification artifacts.

**Fix:** Add the two regression tests described in CR-01 and CR-02. Each should assert the
correct non-green outcome and run with both component sort orderings for CR-01.
