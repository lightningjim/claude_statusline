## Pattern Assignments

### 1. `_derive_claude_status` Rule 3 — resolved-vs-red branch (derivation, transform)

**Analog:** Rule 1 incident loop (≈:1771-1815) for "scan incidents and match tracked
components"; Rule 3 itself (≈:1838-1858) for the degraded-component branch being refined.

**D-04/D-05/D-01 land here.** The discretion note in CONTEXT (lines 101-106) leans toward
computing the resolved decision *inside derivation* and threading a `kind="resolved"`
through the cache. Rule order is unchanged (Rule 1 active > Rule 2 maint > Rule 3 degraded).
The change is: inside the Rule-3 degraded branch, before returning red, look for a
**resolved** incident in `summary["incidents"]` that touches this degraded component.

**How Rule 1 already scans + matches incidents (copy this matching shape, lines 1777-1802):**
```python
for inc in incidents_raw:
    if not isinstance(inc, dict):
        continue
    inc_status = inc.get("status", "")
    if not isinstance(inc_status, str):
        continue
    if inc_status not in ("investigating", "identified", "monitoring"):
        continue
    tracked = _tracked_component_names(inc.get("components", []))
    if not tracked:
        continue
    if _is_suppressed(inc.get("id", ""), inc.get("impact", "none"),
                      inc.get("name", ""), dismissals, cfg):
        continue  # skip — suppressed
    triggered_incidents.append(inc)
```
For resolved detection, the new scan is the same shape but with the **inverted status
filter**: `if inc_status != "resolved": continue`, then intersect
`_tracked_component_names(inc.get("components", []))` with the specific degraded
`comp_name` (D-05: only paint green when a resolved incident actually touches *that*
component; "newest resolved wins" per discretion line 96-98).

**The Rule-3 branch being refined (lines 1845-1858) — preserve red as the default:**
```python
for comp_name in sorted(_CLAUDE_TRACKED_COMPONENTS):  # deterministic order
    status = comp_status.get(comp_name, "operational")
    if status in non_operational:
        human_state = _CLAUDE_STATUS_LABELS.get(status, status.replace("_", " "))
        label = f"{comp_name}: {human_state}"
        severity_map = {
            "degraded_performance": "minor",
            "partial_outage":       "major",
            "major_outage":         "critical",
            "under_maintenance":    "maintenance",
        }
        severity = severity_map.get(status, "minor")
        return {"severity": severity, "label": label, "kind": "degraded"}
```
**Extension shape (D-05):** when a matching resolved incident is found for `comp_name`,
return `{"severity": "resolved", "label": <resolved incident title>, "kind": "resolved"}`
(label gets the `resolved:` prefix at render, not baked into the cache — keep parity with
how active titles are stored RAW). With **no matching resolved incident**, fall through to
the existing red return unchanged. D-06 (muted incident → no green and **no red fallback**)
is enforced at render via `_is_suppressed`, but the resolved incident considered here must
itself respect suppression: reuse the same `_is_suppressed(...)` gate Rule 1 uses (lines
1794-1801) so a muted resolved incident never produces the green item.

**Return-dict contract (must match existing):** every non-None return is
`{"severity": str, "label": str, "kind": str}` (line 1715). Add `"resolved"` as a new
`kind` and a new `severity` token; do not add extra keys (cache/test shape stability).

**Never-raises:** whole body already wrapped in `try/except: return None` (lines 1737/1863).
A resolved-detection failure must degrade to the existing red return, not raise
([[statusline-omit-not-fake]]).

**Analog C — D-06 render-time mute re-check (lines 3584-3628), reuse VERBATIM.** This is the
Phase-7 Plan-04 block that re-runs `_is_suppressed` over cached `tracked_incidents`:
```python

    dismissals = read_dismissals(_DISMISSALS_PATH)
except Exception:
    dismissals = {}
if not isinstance(dismissals, dict):
    dismissals = {}

raw_tracked = sec.get("tracked_incidents")
tracked_incs = raw_tracked if isinstance(raw_tracked, list) else []

            break
```

**D-06 is the critical interop:** a degraded component explained *only* by a muted incident
must stay **quiet and NOT fall back to red**. Today, if the surviving-incident loop finds
nothing it returns `None` only when `tracked_incs` is non-empty; a `degraded`/`resolved`
baked item with empty `tracked_incs` falls through to the baked path (lines 3628-3629).
Planning must ensure a **resolved baked item is exposed to this suppression re-check** —
i.e. the resolved incident must be present in cached `tracked_incidents` (see #5) so that
when it is muted, `surviving_inc is None` → `return None`, and a baked `kind == "resolved"`
is never rendered green for a muted incident. Likewise a `kind == "degraded"` (red) item
whose only explanation is a muted incident must resolve to `None`, not red — confirm the
fall-through guard at planning so D-06 ("muting wins in every state") holds.

**Never-raises:** body already inside `try/except: return None` (lines 3550/3673).

**Analog B — `live_ids` prune block (lines 2103-2133).** This currently keys `live_ids` off
**unresolved** incidents only (lines 2119-2127), and the WR-03 comment (lines 2113-2118)
explains *why*: a resolved incident lingers in the feed for days, so including it would
defer pruning a dismissal that `--status-incidents` already reports as stale.
```python
for _inc in (summary.get("incidents", []) or []):
    if isinstance(_inc, dict):
        _status = _inc.get("status", "")
        if _status not in ("investigating", "identified", "monitoring"):
            continue          # resolved incident — do not keep in live_ids
        _id = _inc.get("id", "")
        if isinstance(_id, str) and _id:
            live_ids.add(_id)
```
**Interop with D-06 — this is the subtle one.** D-06 says a muted incident stays muted
**through its resolved phase**. But this prune drops the dismissal as soon as the incident
leaves the unresolved set (i.e. when it resolves). If the dismissal is pruned at resolution,
the resolved incident would re-surface green (no longer muted) — violating D-06.
**Planning must reconcile:** while a resolved incident is still being *surfaced* (its
tracked component is still degraded, D-01 lifecycle), its dismissal must NOT be pruned, so
`live_ids` (or the prune predicate) likely needs to include resolved-but-still-degraded
ids. This is the most important integration risk in the phase — flag it explicitly in the
plan. Keep the prune block's own `try/except` (lines 2110/2132) so a store error never
affects the fetch (D-10).

## Shared Patterns

### Suppression (apply to #1, #3, #5, #6) — reuse `_is_suppressed` VERBATIM
**Source:** `_is_suppressed` (≈:1579-1700); single impact ordering `_CLAUDE_IMPACT_RANK`
(line 1559). D-06 = "muting wins in every state." Resolved items pass through the *same*
`_is_suppressed` gate as active items — no second suppression path, no duplicated ordering.
A resolving incident has *falling* impact, so the escalation re-surface (D-03 of Phase 7,
lines 1679-1694) never voids a dismissal on the resolved path — the dismissal stands.

### Never-crash / omit-not-fake (apply to ALL)
**Source:** every builder/derivation wraps its body in `try/except` and returns
`None`/`[]`/safe-default (`_derive_claude_status` :1737/1863; `_claude_status_segment`
:3550/3673; `_collect_tracked_incidents` :2014/2046; `fetch_claude_status` :2073/2166;
`_claude_status_color` :1517/1529; printer :501/573). A resolved-detection failure must
degrade to existing behavior (red Rule 3 / silent omit), never raise, never fabricate
([[statusline-omit-not-fake]]).

### RAW-in-cache, sanitize-at-edge (apply to #5, #6, #3, #7)
**Source:** titles stored RAW in cache/`tracked_incidents` (:2040, :2065-2066); the verbatim
ANSI-strip + width-bound sanitizer is applied only at render (`_claude_status_segment`
:3661-3664) and at print (`_status_incidents` printer `_sanitize` :529-533). The `resolved:`
prefix is a render/print concern, not a cached value.

### Render path is cache-only + cheap (apply to #3, #7)
**Source:** the segment and the printer read `read_cache(_CACHE_PATH)` and
`read_dismissals(_DISMISSALS_PATH)` only (:3562/3585, :504/printer) — never the network.
The resolved branch must stay equally cheap and failure-tolerant; all live signal comes
from the cached `summary`-derived payload written by the detached `--refresh` child.

### Test plumbing (apply to test changes)
**Source:** `_load_script_module` (:36), `_load_fixture` (:44), JSON fixtures in
`tests/fixtures/status_*.json`; render tests patch `_CACHE_PATH` + `_DISMISSALS_PATH` and
build a section via `_fresh_section(...)` / `_segment(...)` helpers (:1186-1238). New tests
mirror these exactly:
- new fixture e.g. `status_resolved_degraded.json` (resolved incident + still-degraded
  tracked component) → derivation `kind == "resolved"`, render contains `GREEN`,
  `_NF_GSD_DONE` (nerd) and `resolved:` substring.
- degraded component with **no** matching resolved incident → red (unchanged) regression.
- component operational → quiet (`None`).
- active incident outranks resolved (Rule 1 > Rule 3); maintenance outranks resolved-degraded
  (Rule 2 > Rule 3) — assert kind ordering.
- muted resolved incident → render returns `None`, NOT red (D-06) — use the existing
  `_fresh_section(... tracked_incidents=[resolved_inc])` + dismissal-store pattern
  (:1199-1212).
- `--status-incidents` STATE column shows `resolved`.
- malformed/garbage feed → no crash (mirror `test_*_does_not_raise`, :331-373).
