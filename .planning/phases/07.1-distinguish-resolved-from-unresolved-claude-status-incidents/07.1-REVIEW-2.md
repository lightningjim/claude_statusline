## Critical Issues

### CR-01: Muted-incident degraded component still masks a later genuinely-unexplained RED outage (order-dependent)

**File:** `claude-statusline.py:1960-1976` (Rule 3 early-return branch in `_derive_claude_status`)

**Issue:**
The CR-01 fix correctly keeps scanning past *resolved-explained* degraded
components (it stashes `resolved_result` and continues), so a resolved-green
verdict can no longer mask a later unexplained RED. However, the loop still
**returns immediately** for any degraded component whose `best_resolved is None`
— including the case where the component is "explained" solely by a *suppressed*
(dismissed / keyword-muted) incident. In that case `muting_incident_id` is set,
the function returns a `degraded` verdict bound to that muted id, and the
render-time mute re-check turns the whole segment into `None`. Any
later-sorted, genuinely-unexplained degraded component (a real outage with **no
incident at all**) is never reached.

This is the same order-dependent truth-telling violation CR-01 was meant to kill
(`[[statusline-omit-not-fake]]`, D-05): a real RED outage is hidden purely
because of component sort order.

Reproducing input (verified by direct execution):

```python
summary = {
  "components": [
    {"name": "Claude Code", "status": "partial_outage"},  # sorts FIRST, muted-explained
    {"name": "claude.ai",   "status": "partial_outage"},  # sorts LAST, genuinely unexplained
    {"name": "Claude Cowork", "status": "operational"},
  ],
  "incidents": [
    {"id": "m1", "name": "Mythos noise", "status": "investigating",
     "impact": "minor", "updated_at": "2026-06-18T12:00:00Z",
     "components": [{"name": "Claude Code"}]},
  ],
  "scheduled_maintenances": [],
}
cfg = {"claude_status": {"filter_enabled": True, "ignore_title_patterns": ["Mythos"]}}
# derive -> {"kind":"degraded","incident_id":"m1","component":"Claude Code", ...}
# render -> None   (claude.ai's real partial_outage is MASKED)
```

Reversing which component is muted (muted incident on `claude.ai`, `Claude Code`
genuinely unexplained) renders RED correctly — proving the masking is purely
sort-order-dependent, exactly the CR-01 class of bug.

**Fix:**
Treat a muted-incident-only degraded component the same way resolved-explained
components are treated: do not early-return on it. Defer it (remember the first
muted-degraded result) and keep scanning so a later genuinely-unexplained
component can still win RED. Only emit the muted-degraded (→render `None`)
verdict if NO unexplained component is found. Sketch:

```python
muted_result = None   # alongside resolved_result, before the loop
...
if best_resolved is None:
    if muting_incident_id is not None:
        # muted-explained: defer, do NOT return — a later unexplained comp must win
        if muted_result is None:
            muted_result = {
                "severity": severity_map.get(status, "minor"),
                "label": f"{comp_name}: {_CLAUDE_STATUS_LABELS.get(status, status.replace('_',' '))}",
                "kind": "degraded",
                "incident_id": muting_incident_id,
                "component": comp_name,
            }
        continue
    # genuinely unexplained (no incident at all) -> RED wins immediately
    return { ... "incident_id": None ... }
...
# after the loop, precedence: unexplained RED already returned;
# then resolved-green, then muted (renders None):
if resolved_result is not None:
    return resolved_result
if muted_result is not None:
    return muted_result
return None
```

(Decide resolved-vs-muted precedence deliberately — but a genuinely-unexplained
RED must outrank both, and it currently does not when it sorts after a
muted-explained component.) Add a regression test mirroring
`test_cr01_unexplained_degraded_wins_over_resolved_*` but with the early component
muted instead of resolved, asserting `kind=="degraded"`, `incident_id is None`,
and `component=="claude.ai"`.

## Info

### IN-01: CR-01 multi-component test coverage omits the muted-incident masking case

**File:** `tests/test_claude_status.py:512-599`

**Issue:**
The WR-03 regression tests for CR-01 (`_multi_component_summary` +
`test_cr01_unexplained_degraded_wins_over_resolved_*`) only exercise the
*resolved-explained* masking path. They do not cover a degraded component
explained by a *muted/dismissed* incident sorting ahead of a genuinely-unexplained
one — which is why the surviving masking bug in CR-01 was not caught. The
`_multi_component_summary` helper hardcodes a single resolved incident and never
adds a suppressed/active one.

**Fix:**
Add a test that places a muted (dismissed or keyword-filtered) incident on the
alphabetically-first degraded component and a genuinely-unexplained degraded
state on a later component, asserting the verdict is the RED unexplained
component (and, end-to-end, that `_claude_status_segment` renders non-None for
the real outage). This guards the fix for CR-01 above.
